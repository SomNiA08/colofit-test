SECTION 01

사전 준비 사항

개발을 시작하기 전에 아래 도구와 계정이 필요합니다.

필수 도구

Claude Code CLI - Anthropic 공식 CLI
Node.js 20+ - Next.js 실행
Python 3.13+ - FastAPI 실행
Git - 버전 관리

필요한 계정/API 키

네이버 개발자 - 쇼핑 API 키
Google AI Studio - Gemini API 키
Supabase - PostgreSQL DB
Vercel - 프론트 배포
Railway - 백엔드 배포

Claude Code 설치

아직 설치하지 않았다면 터미널에서 아래 명령어를 실행하세요.

Terminal
npm install -g @anthropic-ai/claude-code # 설치 확인 claude --version
GitHub Repository

프로젝트 저장소: https://github.com/2000silpeed/colorfit.git
기획서, 디자인 시스템, CLAUDE.md, TASK.md, rules가 모두 포함되어 있습니다.
git clone 후 바로 Claude Code를 실행하면 됩니다.

SECTION 02

프로젝트 문서 구조 이해하기

프로젝트 폴더에는 6개의 핵심 문서가 있습니다. 각각의 역할을 이해하세요.

colorfit/ ├── ColorFit_상세기획서_v1.3.md ← 제품 설계 전체 (3,300줄+). 읽기 참조용 ├── DESIGN.md ← 디자인 시스템. UI 만들 때 반드시 참조 ├── CLAUDE.md ← Claude Code가 자동으로 읽는 프로젝트 규칙 ├── TASK.md ← 체크리스트. 진행 상황 추적 └── .claude/ └── rules/ ├── frontend.md ← .tsx 파일 작업 시 자동 로드 ├── backend.md ← .py 파일 작업 시 자동 로드 ├── testing.md ← 테스트 파일 작업 시 자동 로드 └── data-pipeline.md ← 데이터 스크립트 작업 시 자동 로드
문서역할누가 읽나
CLAUDE.md프로젝트 전체 규칙. 기술 스택, 아키텍처, Fallback 전략Claude Code가 항상 자동으로 읽음
.claude/rules/*.md파일별 구체적 코딩 규칙해당 파일 수정 시 자동 로드
TASK.md주차별 체크리스트개발자(당신)가 진행 상황 확인용
DESIGN.md서체, 컬러, 스페이싱 등 디자인 토큰Claude Code가 UI 코드 작성 시 참조
기획서 v1.3제품 설계 전체 (알고리즘, DB 스키마, API 등)필요 시 Claude Code에게 "기획서 섹션 X 참조해" 지시
핵심 포인트

CLAUDE.md와 .claude/rules/는 Claude Code가 자동으로 읽습니다. 별도로 "이 파일 읽어"라고 말할 필요가 없습니다. 프로젝트 폴더에서 claude를 실행하면 자동 적용됩니다.

SECTION 03

Claude Code 시작하기

프로젝트 폴더에서 Claude Code를 실행하고 작업을 지시하는 방법입니다.

1

GitHub에서 프로젝트 클론

터미널을 열고 프로젝트를 클론합니다. 기획서, 디자인 시스템, rules가 이미 포함되어 있습니다.

git clone https://github.com/2000silpeed/colorfit.git cd colorfit
2

환경 설정

.env.example을 복사해서 .env를 만들고, API 키를 채워 넣습니다.

cp .env.example .env # .env 파일을 열어서 실제 API 키를 입력하세요
3

Claude Code 실행

claude 명령어를 입력하면 대화형 세션이 시작됩니다.

claude # Claude Code가 시작되면 CLAUDE.md를 자동으로 읽고 # 프로젝트 컨텍스트를 이해합니다. # 이제 자연어로 작업을 지시하면 됩니다.
4

작업 지시하기

Claude Code에게 무엇을 만들지 자연어로 지시합니다. 구체적일수록 좋습니다.

Claude Code 프롬프트 예시
# 좋은 예시 (구체적) "TASK.md의 W1 Lane B를 진행해줘. Next.js 15 프로젝트를 frontend/ 디렉토리에 초기화하고, TailwindCSS 4.0과 Framer Motion을 설치해줘. DESIGN.md의 디자인 토큰을 CSS variables로 세팅해줘." # 나쁜 예시 (막연함) "프론트엔드 만들어줘"
5

결과 확인 + TASK.md 체크

Claude Code가 작업을 완료하면, 결과를 확인하고 TASK.md의 체크박스를 확인합니다.

# TASK.md가 업데이트되었는지 확인 cat TASK.md | head -50 # 실제 파일이 생성되었는지 확인 ls frontend/ backend/
SECTION 04

2세션 병렬 개발 전략

터미널 2개에서 Claude Code를 동시에 실행하여 개발 속도를 2배로 높이는 방법입니다.

터미널 1 (Lane A: 데이터) 터미널 2 (Lane B: 인프라) ┌──────────────────────┐ ┌──────────────────────┐ │ backend/scripts/ │ │ frontend/ │ │ backend/data/ │ │ backend/app/ │ │ │ │ docker-compose.yml │ │ 12톤 팔레트 JSON │ │ Next.js 초기화 │ │ 네이버 API 수집 │ │ FastAPI 초기화 │ │ 전처리 파이프라인 │ │ DB 스키마 적용 │ │ 코디 조합 생성 │ │ 배포 설정 │ └──────────────────────┘ └──────────────────────┘ │ │ └──────────┬────────────────┘ ▼ 둘 다 완료 후 합치기 git add . && git commit
충돌 방지 규칙

두 세션이 같은 파일을 동시에 수정하면 Git 충돌이 발생합니다. 각 세션이 작업하는 디렉토리가 겹치지 않도록 주의하세요. 특히 TASK.md는 한쪽 세션에서만 업데이트하게 지시하세요.

실전 실행 방법

터미널 1 프롬프트

cd colorfit # clone한 프로젝트 폴더 claude # 아래를 입력: "TASK.md의 W1 Lane A: 데이터 파이프라인 전체를 순서대로 진행해줘. 12톤 팔레트 JSON 생성부터 시작해서, 네이버 쇼핑 API 수집 스크립트, 전처리, 코디 조합 생성까지. 각 Task 끝나면 TASK.md 업데이트해줘."

터미널 2 프롬프트

cd colorfit # clone한 프로젝트 폴더 claude # 아래를 입력: "TASK.md의 W1 Lane B: 인프라 셋업을 진행해줘. frontend/ 디렉토리에 Next.js 15 프로젝트 초기화, backend/ 디렉토리에 FastAPI 프로젝트 초기화. DESIGN.md의 디자인 토큰을 CSS variables로 세팅해줘. TASK.md는 업데이트 하지 말고 나한테 알려줘."

주차별 병렬 조합

주차터미널 1터미널 2주의
W1 Lane A: 데이터 파이프라인 Lane B: 인프라 셋업 완전 독립. 충돌 없음
W2 Lane C: 추천 엔진 (백엔드) Lane D: 온보딩 + 피드 UI schemas/ 먼저 합의
W3 가격비교 백엔드 가격비교 + 마이페이지 UI W3 금 Fallback 판단
W4~5 순차 통합 (1세션 권장) 통합 테스트 + 버그 수정
SECTION 05

주차별 구현 가이드

각 주차에 Claude Code에게 어떻게 지시해야 하는지 구체적인 프롬프트 예시입니다.

W1 데이터 + 인프라 (3/24~3/28)

첫 주가 가장 중요합니다. 데이터가 없으면 추천 엔진을 만들 수 없습니다.

W1 시작 전 직접 해야 할 것 (Claude Code가 못 하는 것)

1. 네이버 개발자 센터에서 애플리케이션 등록 → Client ID/Secret 발급
2. Supabase 프로젝트 생성 → Database URL 확보
3. Google AI Studio에서 Gemini API 키 발급
4. 위 키들을 .env 파일에 저장

W1 프롬프트 예시 — 하나씩 시키기
"Task 1.1을 진행해줘. 완료되면 TASK.md 업데이트해줘." # Task 1.1 끝나면: "Task 1.2를 진행해줘." # 또는 여러 개를 한번에: "Task 1.1부터 1.4까지 순서대로 진행해줘. 각 Task 끝날 때마다 TASK.md 업데이트해줘."
W1 인프라 — 터미널 2에서
"Task 1.12부터 1.14까지 진행해줘. TASK.md는 터미널 1에서 업데이트하니까 건드리지 마."

W2 추천 엔진 + 온보딩 UI (3/31~4/4)

백엔드에서 추천 로직을, 프론트에서 온보딩 화면을 동시에 만듭니다.

W2 프롬프트 예시 (추천 엔진)
"Task 2.1을 진행해줘. PCF 스코어링 함수와 pytest 테스트를 만들어줘. 완료되면 TASK.md 업데이트해줘." # Task 2.1 끝나면: "Task 2.2를 진행해줘."
W2 프롬프트 예시 (온보딩 UI)
"Task 2.14를 진행해줘. 온보딩 Step 1 성별 선택 화면. DESIGN.md 토큰을 적용해줘." # Step 1 끝나면: "Task 2.15를 진행해줘."

W3~5 가격비교, 결정 지원, 폴리싱

W3 금요일이 Fallback 판단 시점입니다. TASK.md를 확인하고 일정이 밀리면 기능을 2차로 미루세요.

Fallback 순서 (밀릴 때 순서대로 미루기)

1순위: A vs B 비교 → 2차
2순위: Top Pick + One-shot → 2차
3순위: 가격비교 → 2차 (외부 링크만 유지)
4순위: 스코어링 5축 → 3축(PCF+OF+SF)으로 단순화

절대 미루지 않는 것: 온보딩 → 코디 피드 → 추천 이유 → save/dislike → 외부 링크

SECTION 06

자주 쓰는 Claude Code 명령어

Claude Code 세션 안에서 사용하는 슬래시 명령어들입니다.

명령어설명언제 사용
/compact대화 내용을 압축하여 컨텍스트 절약대화가 길어졌을 때 (30분+ 작업 후)
/clear대화 내역 초기화새로운 작업 시작할 때
/help사용 가능한 명령어 목록처음 사용할 때
! 명령어셸 명령어를 직접 실행! npm run dev 등 서버 실행

효과적인 프롬프트 작성법

핵심 원칙: TASK.md의 항목을 하나씩 시키세요

TASK.md에 이미 상세한 Step이 나뉘어 있습니다. 기획서를 직접 참조할 필요 없이, TASK.md의 체크리스트 항목을 하나씩 지시하면 Claude Code가 기획서를 알아서 참조합니다.

좋은 프롬프트

방법 1: TASK.md 항목 지정
"TASK.md의 W1 Lane A에서
'12톤 팔레트 JSON 데이터 생성' Task를 진행해줘.
완료되면 TASK.md 업데이트해줘."

방법 2: 여러 항목 한번에
"TASK.md의 W1 Lane B 전체를 순서대로 진행해줘.
각 Task마다 TASK.md 업데이트해줘."

방법 3: 맥락이 필요할 때만 기획서 참조
"TASK.md의 '하이브리드 분류' Task를 진행해줘.
기획서 섹션 5.4.1에 상세 설계가 있어."

나쁜 프롬프트

X "프론트엔드 만들어줘" → 범위가 너무 넓음

X "기획서 읽고 전부 구현해줘" → 3,300줄을 한번에 읽으면 비효율

X "W1~W3 전부 해줘" → 한 세션에 너무 많은 작업

프롬프트 템플릿

"Task [번호]를 진행해줘. 완료되면 TASK.md 업데이트해줘."

이것만으로 충분합니다. 각 Task에 참조할 기획서 섹션, 파일 경로, 세부 항목이
이미 적혀 있으므로 Claude Code가 알아서 찾아갑니다.

범위 조절:
작게: "Task 2.1을 진행해줘" (1개 함수)
보통: "Task 2.1부터 2.5까지 진행해줘" (5축 스코어링 전체)
크게: "W1 Lane A 전체를 진행해줘" (데이터 파이프라인 전체)

SECTION 07

꿀팁과 주의사항

1. 기획서 섹션을 정확히 지정하세요

기획서가 3,300줄이라 Claude Code에게 "기획서 읽어"라고 하면 전부 읽느라 시간이 걸립니다. "기획서 섹션 5.5 참고해서"처럼 특정 섹션을 지정하면 훨씬 빠르고 정확합니다.

2. 한 번에 너무 많은 것을 시키지 마세요

"온보딩 5개 Step + 코디 피드 + 상세 페이지 + 저장 목록 전부 만들어줘"보다 한 화면씩 시키는 게 품질이 훨씬 좋습니다. 완료 확인 후 다음 화면으로.

3. /compact를 자주 쓰세요

Claude Code는 대화가 길어지면 앞부분을 잊기 시작합니다. 30분 이상 작업하면 /compact로 압축하세요. 컨텍스트를 절약하면서 CLAUDE.md의 규칙은 계속 유지됩니다.

4. .env 파일은 직접 만드세요

API 키, DB 비밀번호 같은 민감한 정보는 Claude Code에게 시키지 마세요. .env.example을 참고해서 직접 .env 파일을 만드세요.

# .env.example (이 파일은 Git에 올라감) NAVER_CLIENT_ID=your_client_id_here NAVER_CLIENT_SECRET=your_secret_here GEMINI_API_KEY=your_gemini_key_here DATABASE_URL=your_supabase_url_here # .env (이 파일은 Git에 올리면 안 됨!) NAVER_CLIENT_ID=abc123...

5. 결과물을 반드시 직접 확인하세요

Claude Code가 만든 코드가 실제로 동작하는지 직접 실행해서 확인하세요.

# 프론트엔드 실행 cd frontend && npm run dev # → http://localhost:3000 에서 확인 # 백엔드 실행 cd backend && python -m uvicorn app.main:app --reload # → http://localhost:8000/docs 에서 API 문서 확인 # 테스트 실행 cd backend && pytest -v cd frontend && npx vitest run

6. 이것만은 절대 하지 마세요

X .env 파일을 Git에 커밋하지 마세요
X Claude Code에게 API 키를 직접 알려주지 마세요
X 두 세션이 같은 파일을 동시에 수정하게 하지 마세요
X 확인 없이 코드를 프로덕션에 배포하지 마세요
